# UI変更ルール

更新日: 2026-07-22

## 目的

スマホで見やすく操作しやすい画面へ改善しながら、安定している業務ロジック、API、DB、計算結果を壊さない。

## 絶対ルール

UI変更で変更してよい範囲は、レイアウト、配置、CSS、画面コンポーネントの分割、表示方法に限定する。

UI変更として、以下を変更してはならない。

- `src/`配下のサーバー処理
- DBスキーマ、マイグレーション、保存形式
- APIのURL、リクエスト、レスポンス
- 集計・計算処理
- 在庫・原価・売上などのデータ経路
- 登録・更新・使用停止を行う既存イベント処理

業務仕様上これらの変更が必要な場合は、UI変更と混ぜず、独立した仕様変更としてレビュー、実装、テスト、コミットする。

## 画面単位の手順

1. 変更前のAPI・登録・更新動作をテストして固定する。
2. 対象画面だけのHTML/CSS・表示方法を変更する。
3. PCとスマホで横スクロール、固定要素の重なり、ボタン操作を確認する。
4. 登録、更新、使用停止、戻る、メニュー遷移を確認する。
5. 全自動テストを実行し、対象外画面へ影響がないことを確認する。
6. 対象画面ごとに独立コミットする。

## 実施順

1. マスタ管理
2. 検品入力
3. 工程実績入力
4. 製造・完成、納品準備、出荷・納品
5. 在庫、棚卸・調整
6. ダッシュボード、詳細分析

一度に全画面を書き換えず、各画面の確認完了後に次へ進む。
