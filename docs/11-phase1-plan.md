# ⑪ Phase1実装計画

## 完了条件

- 社長がれんこん／青果、日／週／月を切り替え、前日判断に必要な指標を確認できる
- 管理者が入荷、工程、製造、出荷、在庫、マスタを運用できる
- 原料、加工済れんこん、完成品、フレーバー、包材の在庫が元帳一致する
- LINE開始・終了から工程別・人別・人別工程工数を確認できる
- 個人別加工表の使用量・破棄・出来高から人別歩留まりを確認できる
- 監査、バックアップ、日次確認が動作する
- 検品→原料在庫→加工→中間／完成在庫→袋詰め→出荷→納品日売上→在庫反映を、れんこんチップス・ねぎ・きゅうりの各フローで1ロット通して確認できる
- 検品画面は紙帳票順の10項目（確定追加の破棄理由を含む）で入力でき、原料在庫は実測数量－検品破棄数量で計上される
- Phase1ロットは内部自動採番・DB保存・在庫紐付けまで動作する

## 実装ゲート

### Gate 0 — 設計レビュー（現在）

手書きイメージ、部門マッピング、週開始、フレーバー共通原料の配賦方法を確認します。月給者の管理時間単価、チップス全体への労務費集約、青果トップ、重量g統一は追加補足で確定済みです。共通原料配賦は拡張構造を先に用意し、Phase1全体の開始を止めません。**設計レビュー承認前は実装しません。**

### Gate 1 — 基盤とDB

Node.js構成、SQLiteスキーマ、マイグレーション、監査、バックアップを構築します。社内限定利用のPhase1ではログイン・権限制御を実装しません。

### Gate 2 — マスタと移行

過去取引・在庫履歴は移行せず、必要な商品・フレーバー・工程・従業員・取引先等のマスタだけを旧ID付きで確認移行します。列版混在、空品目ID、商品重複、部門不整合は確認一覧へ出し、承認済み行だけを移行します。判断不能は保留し、開始在庫は初回棚卸で登録します。

### Gate 3 — 最小業務入力

検品→工程→製造・袋詰め→出荷の順に、在庫元帳とDB取引を含め実装します。検品は実測－検品破棄を原料在庫へ計上し、破棄理由を必須とする。れんこん水煮は販売せず中間在庫、ねぎ・きゅうり加工は袋詰めまでを含む。まとめ製造入力もここで実装します。

### Gate 3.5 — LINE工数

現場の作業開始・終了、工程選択、署名検証、重複排除、未終了確認、管理者の理由付き訂正を実装します。LINEから数量、歩留まり、入荷、在庫、出荷は登録させません。自動取得が利用可能になった時点で個人加工実績画面の時間入力を削除し、LINEセッションから算出します。

### Gate 4 — 社長ダッシュボード

静的モック承認済みの配置へ日・週・月集計を接続し、判断用トップ、工程・人員・商品採算・在庫の詳細、確認状態を実装します。

### Gate 5 — 運用開始準備

棚卸、在庫調整・履歴、最低限のマスタ管理、日次締め、操作説明、移行リハーサル、並行稼働、1ロット総合試験、復元試験を行います。CSV入出力はPhase2へ送ります。

## 試験

Phase1原価の完了条件は、納品書原価、製造消費、納品準備資材、直接外注費から出荷時直接原価・粗利・粗利率を固定保存できること、およびチップス加工・袋詰め・納品準備の労務費をSKUへ配賦せず部門共通労務費として人別・工程別に確認できることとする。

- 歩留まり境界: 投入0、後数量超過、副産物、重量g固定、既存kg値の移行変換
- 在庫: FIFO、翌日持越し、不足時ロールバック、取消・訂正、棚卸
- 工数: 日跨ぎ、休憩、複数人、重複セッション
- 個人加工: 1人1行、使用量0、破棄、出来高、その他差異、LINE時間紐付け、複数人合計
- 画面分離: ダッシュボード内では更新不可。三本線メニューから管理入力へ切り替える
- 集計: 同一原本から日・週・月の合計が一致、取消除外、未入力表示
- 移行: マスタ件数・ID・在庫残・主要期間KPIを旧システムと照合
- 運用: 日次バックアップ、月次復元確認、監査検索

本番は日次バックアップ、30日保持、月1回復元確認を必須とし、秘密値はRender環境変数へ保存してGitHubへ登録しない。

## 開始優先のスコープ制御

Phase1にはLINE開始・終了による工数取得を含めます。売掛入金・消込、資金繰り連携、OCR、AI、CSV入出力、PDF帳票、複数会社、高度な原価分析、ロット検索・追跡・逆引きはPhase2とします。LINEで実装する外部イベント受信箱をPhase2連携でも再利用します。
