## 結論

このExcelは既存GASアプリが利用しているスプレッドシートDBの実体に近く、コードだけでは分からなかった運用状態を確認できました。27シートありますが、多くの分析・売上・製造シートはヘッダーのみで、現時点の蓄積は仕入、買掛、ログを中心としています。

そのまま新DBへ一括移行するのは危険です。列追加がヘッダーへ反映されていない箇所、旧列定義と新しい書込順の不一致、品目ID欠落、部門割当の不整合、g/kg混在が確認できるため、検証・対応表・例外確認を挟む必要があります。

## データ品質上の確認事項

### 1. T_工程のスキーマずれ

工程行には歩留まりに続いて「投入量−加工後量」と一致する値と、その割合に一致する値がありますが、見出しは旧版の「作業時間」「作業人数」のままです。また、「登録日時」見出しの列には振分先らしき値が入り、実際の日時は右側の見出しなし列へずれています。書込処理と列定義が途中で変わった可能性があります。列位置だけで自動移行すると、ロス量を作業時間、ロス率を作業人数として扱う誤変換が起こります。

したがって、このExcelのT_工程から既存工数を自動移行しません。各行を登録時点のコード版・LOG・担当者明細・勤怠と照合し、復元できる項目だけを採用します。

### 2. T_仕入の状態列

最終列に状態らしき値がある行と空の行がありますが、ヘッダーが空です。空を有効と扱うかは、登録時期・取消ログと照合して決めます。

### 3. 商品と部門

青果加工品がれんこんチップス製造部門に紐付く例があります。入力ミス、共通商品、設計上の部門統合のどれかを確認し、名称から自動修正しません。

### 4. 商品重複候補

同じ商品名・内容量に見える複数IDがあります。過去参照があるため勝手に統合せず、統合先IDと旧ID対応表を作ります。

### 5. 在庫の品目ID欠落

加工済在庫などに商品IDが空の行があります。新DBでは品目ID必須とし、既存行は工程記録・部門・登録日時から候補を出して管理者確認します。

### 6. g/kg混在

在庫・工程にgとkgが混在します。新DBの重量正本はgで統一するため、既存kg値は移行処理でgへ明示変換し、元値・元単位・換算結果を監査記録へ残します。新規入力・画面表示ではkgを使用しません。単位なし・換算不能は取込エラーにします。

### 7. 集計値の再利用

A_工程分析・A_人別分析などは移行正本にしません。T_工程、T_工程_担当者、T_勤怠から再計算し、旧集計値とは検算にだけ使用します。

## 移行方針

1. 原本Excelを読取専用で保全し、ハッシュを記録する。
2. シート版を識別し、列位置と意味の対応表を作る。
3. マスタを検証し、旧ID→新ID対応表を確定する。
4. g/kg、部門、商品重複、空IDをエラー一覧として管理者確認する。
5. T_系原本を優先して取込み、A_系は開始在庫と検算に限定する。
6. 日別・部門別の入荷、工程前後量、歩留まり、在庫を旧データと照合する。
7. 不明行は推測で登録せず、保留行として残す。
