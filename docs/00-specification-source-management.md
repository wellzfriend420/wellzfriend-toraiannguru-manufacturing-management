# 仕様情報源と優先順位

## 設計者の必読順

設計・レビュー・実装に参加する人は、最初に次の順番で資料を読みます。途中の資料だけを参照して仕様を決定してはいけません。

1. 本書「仕様情報源と優先順位」
2. 「社長補足仕様」
3. 「社長手書きメモ」
4. 「個人別加工帳票」
5. 「既存アプリ解析」

この順番は、上位の意図を理解したうえで、画面イメージ、個人別実績の運用、既存資産の詳細へ進むための読解順です。

## 推測による仕様確定の禁止

- 設計途中で、画像、名称、一般的な業務慣行だけを根拠に仕様を確定しません。
- 一意に判断できない内容は「判読確認待ち」として管理します。
- 複数の情報源が異なる内容を示す場合は「競合確認待ち」として管理します。
- 「判読確認待ち」「競合確認待ち」は、ユーザーレビュー後にのみ確定へ変更します。
- 既存アプリを解析する前に、新しい入力項目、集計ロジック、DB構造を独自に決定しません。
- 安全性、法令、データ整合性に関する懸念は推測で解消せず、競合または確認事項として明示します。

## 正式仕様の統合方針

正式仕様は次の3情報源を統合して作成します。

1. 社長の考えをChatGPTとの壁打ちで整理した補足内容
2. 社長手書きメモで明確に示された画面イメージ・考え方
3. 既存製造管理アプリ

内容が競合する場合の優先順位は、**補足内容 ＞ 社長手書きメモ ＞ 既存製造管理アプリ**です。安全性、法令、データ整合性に関わる問題がある場合は、競合として明示し、無断で低い基準へ合わせません。

手書き文字からの推測と既存アプリの実際の運用・データが競合する場合は、推測より実運用・データを優先します。これは、手書きメモで明確になっている社長の意図を既存アプリで上書きするものではありません。

## 過去の運用試行資料

既存アプリ導入前に運用を試みたExcelは、正式仕様の第4情報源にはせず、**補助的な lessons learned 資料**として扱います。現在の操作・計算を上書きする根拠にはしませんが、以前から必要だった管理項目と、継続運用を難しくした可能性のある構造を確認するために使用します。

- 活かすもの: 業務項目、管理単位、工程・在庫・SKU・工数を関連付ける考え方
- 活かさないもの: 多シート入力、固定列の横展開、セル数式連鎖、時間の手入力
- 競合時: 補足仕様、明確な手書き意図、現在の既存アプリ実運用を優先
- 失敗理由: ユーザー確認前に推測で確定しない

## 反映状態

各仕様は次の状態を持ちます。

|状態|意味|
|---|---|
|確定|補足内容で明示され、実装可能な定義になっている|
|暫定|手書きメモまたは既存アプリから読み取れるが、補足確認が必要|
|判読確認待ち|手書き文字や意図を一意に読めない|
|競合確認待ち|複数情報源の内容が異なる|
|将来|Phase2以降に実装する|

## 追記時の更新手順

1. ユーザーが提示した補足内容を、日付と要点付きで最上位仕様として記録する。
2. 対応する手書きメモ項目と既存機能を紐付ける。
3. 優先順位に従って、画面、DB、API、集計、実装計画を更新する。
4. 以前の暫定仕様が変わった場合、削除せずCHANGELOGへ変更理由を残す。
5. 実装前レビュー項目の状態を更新する。

補足内容を単なる参考意見として扱わず、社長の意図を整理した上位仕様として扱います。

## 仕様確定の条件

「確定」にできるのは、次のいずれかを満たす項目だけです。

1. 社長補足仕様で明示され、実装可能な定義になっている
2. 既存アプリのコード、データ、画面、実運用から確認でき、上位仕様と競合しない
3. 判読または競合事項について、ユーザーレビューで明示的な回答を得た

設計者による妥当そうな補完、業界標準、画面上の空欄は、確定根拠に含めません。

## 社長Dashboard設計における適用順

社長Dashboardの工程・表示項目を設計するときは、上記の正式仕様統合方針を守ったうえで、次の順に照合します。

1. 食品加工管理帳票に記載された実際の工程と入力項目
2. 社長Dashboard設計修正の補足・確定指示
3. 社長手書きメモの確認順と画面イメージ
4. 既存アプリの利用可能なデータ構造・集計ロジック

帳票と既存工程マスタの名称・順序が一致しない場合は「競合確認待ち」とし、どちらかへ推測で統一しません。
