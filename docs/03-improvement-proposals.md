# ③ 改善提案

## 優先提案

### 1. 社長画面を最初に受入確認する

実装順は「全入力画面の完成後にダッシュボード」ではなく、社長画面の静的モック → 指標定義合意 → 最小入力 → 実データ集計とします。手書きイメージ提供後、表示順・名称・色・単位を社長本人の言葉に合わせます。

### 2. 前日確定の状態を見せる

朝の数字が入力途中では判断を誤ります。部門・業務日ごとに「未入力／入力中／確認済み」を表示し、管理者が確認済みにする運用を追加します。締め後も管理者は理由付きで再開できます。

### 3. 管理者入力は4つに絞る

Phase1の主要導線を「入荷」「工程実績」「製造実績」「出荷・売上」に絞り、在庫は自動元帳化します。フレーバー・包材の直接消費は製造明細の展開が難しい初期だけ管理者入力を許可します。

### 4. まとめ入力と単品入力を同じ保存形式にする

画面上は明細行を複数追加でき、1行だけなら単品入力になります。DBは製造ヘッダー＋製造明細とし、将来のOCRやLINEも同じAPIへ投入します。

### 5. 「人別歩留まり」の意味を明確にする

既存機能名に合わせ「人別歩留まり」と表示します。新規実績は食品加工管理帳票に基づき1人1行で使用量・破棄量・出来高を保持し、個人出来高÷個人使用量で算出します。既存の親工程配賦値は過去参考に限定します。人員分析へ一元表示し、工程分析との二重表示を避けます。Phase1では段取り・作業効率確認用であり、人事評価へ使わない注記を付けます。

## 入力項目の整理

|提案|内容|
|---|---|
|統合|開始・終了から合計時間を算出し、合計時間の二重入力を廃止|
|統合|商品に既定単位・部門・在庫種別を持たせ、取引時の反復入力を削減|
|自動|売上金額は数量×単価。権限者のみ理由付き上書き|
|自動|ロス量・廃棄量・歩留まり・ロス率|
|追加|入力元（web/line/ocr/import/n8n）、外部ID、取込日時。Phase1 LINEは開始・終了のみ|
|追加|確認状態、確認者、確認日時、訂正理由|
|削除候補|工程の「振分先」自由入力。加工後品目・在庫場所の選択へ置換|
|削除候補|製造の作業人数と担当者IDの併存。担当者明細から人数を算出|

## 運用移行

1. 既存アプリは参照可能なまま維持する。
2. マスタをCSVで新DBへ取込み、件数・ID・有効状態を照合する。
3. 3〜5営業日は管理者が旧新を並行入力し、日次KPIと在庫差を比較する。
4. 社長が表示を承認し、管理者が入力手順を承認した後に新を正本化する。
5. 旧アプリは読取専用化し、保持期間確定後も即時削除しない。

## 非機能改善

- DB取引により、取引登録と在庫移動を一括成功／一括失敗にする
- すべての更新を認証・権限・監査対象にする
- 同じ送信の再試行を冪等キーで重複登録しない
- 日次自動バックアップと月次復元確認を継承する
- 社長APIは閲覧専用かつ必要情報だけ返す
- 集計値には元明細へのドリルダウンを付け、数字の根拠を確認可能にする
