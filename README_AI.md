## 目的

とらいアンぐる既存アプリの業務資産を継承し、Node.js＋SQLiteで製造管理システムを新規開発する。最優先成果は、社長が朝に前日実績を確認し当日の段取りを決めるダッシュボードである。

業務仕様が競合する場合は、ChatGPTとの壁打ちで整理された補足内容、社長手書きメモ、既存アプリの順に優先する。画像の判読推測を確定仕様として扱わない。

## Phase2Aの境界

- 通常のダッシュボードと入力画面は認証なし。
- LINE利用者管理、工数訂正、会社設定のみ管理者PINが必要。
- LINE Webhookはn8nが受け、業務判断は本アプリが行う。
- n8n APIはHMAC署名、時刻制限、イベントID重複防止を必須とする。
- LINE工数の訂正は理由と訂正前後を保存する。
- `line_events`は生受信監査、`external_events`は冪等制御として混同しない。

## 守ること

- 社長ロールは閲覧専用。画面非表示だけでなくAPIで更新を拒否する
- 在庫と取引を同一DB取引で更新し、負在庫を許可しない
- 派生集計を正本にせず、取引・在庫移動・工数原本から再計算する
- 取消・訂正・日次再開は理由と監査を残す
- 新規人別歩留まりは個人加工実績の出来高÷使用量。既存の親工程配賦方式は過去参考に限定する
- データ経路は現場実績→個人別加工実績→工程実績集計→社長Dashboard
- Dashboardトップは判断用、工程・人員・商品採算・在庫の詳細は原因確認用
- Dashboardと集計表は直接編集禁止。原本訂正後に再計算する
- 総合歩留まりはトップへ単独表示せず、工程別歩留まりを表示する
- 工程マスタは労務費配賦単位。Phase1はれんこん5工程、青果は入荷・加工（袋詰め含む）・納品準備・配達納品を初期値とする
- チップス端数は廃棄せず半製品在庫へ持ち越す
- 売上は納品日基準。現金は同日回収済み、売掛は残高・入金予定日・消込を保持する
- 包材単価は仕入金額÷数量、商品包材原価はロット単価×実使用数量
- 月給者の原価配賦用時間単価は月給÷当月平日日数÷8。給与計算には使用しない
- チップス労務費はチップス全体原価へ集約し、SKU・フレーバー・内容量別に配賦しない
- 青果トップは検品実測数量・加工使用量・出来高・加工破棄・歩留まり・出荷量・売上を商品別表示する。収穫量・在庫・取引先別売上は表示しない
- 検品は紙帳票順の受入日、仕入先、品目、箱数、担当者、納品数量、実測数量、破棄数量、破棄理由、備考を原本とし、原料在庫は実測－検品破棄で計上する。箱数換算は禁止
- 水煮は販売せずチップス用中間在庫。Phase1ロットは内部自動採番・保存・在庫紐付けまでで、検索・追跡・逆引きはPhase2
- 重量は内部・入力・API・画面表示ともg固定。kg自動変換なし
- Phase1 LINEは開始・終了だけ。数量・歩留まり・入荷・在庫・出荷をLINE入力させない
- LINE自動取得後、通常の個人加工表に時間入力列を置かない
- Phase1直接原価は原料・包材・フレーバー・直接資材・商品直接外注費。出荷時に内訳・粗利・粗利率を固定保存する
- チップス加工・袋詰め・納品準備の労務費はSKUへ配賦せず、部門共通労務費として人別・工程別に分析する
- OCR・AI・n8nはPhase1へ入れない
- 実データ、認証情報、個人情報をリポジトリへ保存しない
- 既存Excelの列名欠落・旧新スキーマ混在を推測補完して本番移行しない
- コミットb13e06fのmigratePhase2Schemaだけで既存DBが完全移行されると仮定しない
